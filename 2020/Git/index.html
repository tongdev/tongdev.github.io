2 本地库命令行操作

2.1 本地库初始化

  • 命令：git init

  • 效果：在当前目录下生成一个 .git 目录，存放的就是本地库相关的信息，不要删除，也不要随便修改

2.2 设置签名

  • 命令

    • 项目级别 / 仓库级别：仅在当前本地库范围内有效

      git config user.name hellotong_pro

      git config user.email hellotong_pro@qq.com

      信息保存位置：当前项目文件夹/.git/config 中

    • 系统用户级别：登录当前操作系统的用户范围

      git config –global user.name hellotong_glb

      git config –global user.email hellotong_glb@qq.com

      信息保存位置：C:/Users/用户文件夹/.gitconfig

  • 级别优先级

    就近原则：项目级别优先于系统用户级别，二者都有时采用项目级别的签名

    如果只有系统用户级别的签名，就以系统用户级别的签名为准

    二者都没有不允许，一般设置一个系统用户级别的签名就可以了，如果有特殊需求在设置项目级别的签名

2.3 基本操作

2.3.1 状态查看

  • 命令：git status

    查看工作区、暂存区状态

2.3.2 添加

  • 命令：git add [filename]

    将工作区的 “新建 / 修改” 添加到暂存区

2.3.3 提交

  • 命令：git commit -m “commit message” [filename]

    将暂存区的内容提交到本地库，注意 commit message 必须有，更多关于 commit message 的规范可点击查看

2.3.4 查看历史记录

  • 命令：git log

    显示每个提交记录的详细信息

  • 命令：git log –pretty=oneline

    显示每个提交记录在一行

  • 命令：git log –oneline【常用】

    与 git log –pretty=oneline 的区别是只显示前 7 位提交记录哈希值

  • 命令：git reflog【常用】

    可以看到每个版本的历史记录，HEAD@{移动到当前版本需要多少步}

2.3.5 版本前进后退

  • 版本控制的本质

    移动 HEAD 指针，默认 HEAD 指针是指向最新的版本

  • 基于索引值操作【推荐】

    命令：git reset –hard [版本索引值 / 哈希值]

    eg: git reset –hard a6ac99

  • 使用 ^ 符号：只能后退

    命令：git reset –hard HEAD^

    指的就是在当前版本（HEAD）的基础上后退一个版本

    注：一个 ^ 表示后退一步，n 个 ^ 表示后退 n 步

  • 使用 ~ 赴会：只能回退

    命令：git reset –hard HEAD~n

    注：n 表示后退几步 / 几个版本

2.3.6 reset 命令了的三个参数对比

  • –soft 参数

    仅在本地库移动 HEAD 指针

  • –mixed 参数

    在本地库移动 HEAD 指针

    重置暂存区

    但工作区不会有什么变化

  • –hard 参数【推荐】

    在本地库移动 HEAD 指针

    重置暂存区

    重置工作区

2.3.7 删除文件并找回

  • 前提：删除前，文件存在时的状态提交到了本地库

  • 命令：git reset –hard [指针位置]

    删除前操作已经提交到本地库：指针位置指向历史记录

    删除操作尚未提交到本地库：指针位置使用 HEAD

2.3.8 比较文件差异

  • 命令：git diff [文件名]

    将工作区中的文件和暂存区进行比较

  • git diff [本地库中历史版本] 文件名

    将工作区的文件和本地库历史记录比较

  • 不带文件名比较多个文件（比较全部）

2.4 分支管理

2.4.1 什么是分支？

在版本控制中，往往使用多条线同时推进多个任务。

2.4.2 分支的好处

  • 同时并行推进多个功能开发，提高开发效率
  • 各个分支在开发过程中，如果某一个分支开发失败，不会对其他分支有任何影响。失败的分支删除即可。

2.4.3 分支操作

  • 创建分支

    命令：git branch [分支名]

  • 查看分支

    命令：git branch -v

  • 切换分支

    命令：git checkout [分支名]

  • 合并分支

    第一步：切换到接收修改的分支（被合并、添加新内容）上

    命令：git checkout [被合并分支名]

    第二步：执行 merge 命令

    命令：git merge [有新内容分支名]

  • 解决冲突

    冲突的表现：当多个分支操作同一位置，先提交到本地库的分支不会出现问题，后提交的分支就会产生冲突，需要解决冲突。

    冲突的解决
    第一步：编辑文件，删除特殊符号

    第二步：把文件修改到满意的程度，保存退出

    第三步：git add [文件名]

    第四步：git commit -m “日志信息”（注意：此时 commit 一定不能带具体文件名）

3 远程库操作

3.1 创建远程库

在 GitHub / 码云上创建一个远程库

3.2 创建远程库地址别名

  • git remote -v 查看当前所有远程地址别名
  • git remote add [别名] [远程地址（HTTP / SSH 都可）]

3.3 推送

  • git push [别名] [分支名] 推送本地库内容到远程库

3.4 克隆

  • git clone [远程地址]

  • 效果

    完整的把远程库下载到本地

    创建 origin 远程地址别名

    初始化本地库

3.5 邀请团队成员

仓库拥有者向团队成员发送仓库邀请链接，团队成员点击接收邀请即可参与进来

3.6 拉取

推荐使用 fetch 和 merge 两步操作

  • pull = fetch + merge

  • git fetch [远程地址别名] [远程分支名]

  • git merge [远程仓库地址别名 / 远程分支名]

    先拉取远程库内容到本地远程分支上，可以使用 git diff 命令查看一下代码差异。

    所取回的更新，在本地主机上要用”远程主机名/分支名”的形式读取。比如origin主机的master，就要用origin/master读取，使用 git checkout origin/master 命令读取。

    发现没有问题时，使用git merge命令或者git rebase命令，在本地分支上合并远程分支。

  • git pull [远程库地址别名] [远程分支名]

3.7 解决冲突

如果不是基于 GitHub 远程库的最新版所做的修改，不能推送（push），必须先拉取（pull）

拉取下来后如果进入冲突状态，则按照 ”分支冲突解决“ 操作解决即可。

3.8 跨团队协作

  1. 先 Fork 拥有者的远程库成为自己的
  2. 本地修改然后推送到自己的远程库

  3. 创建 Pull Request（PR）给原仓库拥有者

  4. 原仓库拥有者进行代码审核，审核通过后合并代码

3.9 SSH 免密登录

  • 进入当前用户的根目录

    cd ~

  • 删除 .ssh 目录

    rm -rvf .ssh

  • 运行命令生成 .ssh 密钥目录

    ssh-keygen -t rsa -C 任意文本

  • 进入 .ssh 目录查看文件列表

    cd .ssh

    ls -lA 查看隐藏文件

  • 查看 id_rsa.pub 公钥文件内容

    cat id_rsa.pub

  • 复制 id_rsa.pub 文件内容，登录 GitHub，点击用户头像→Settings→SSH and GPG keys

  • New SSH Key，起一个名字，粘贴刚刚复制的公钥信息

  • 回到终端创建远程地址别名

    git remote add origin_ssh [远程仓库 SSH 地址]

  • 推送文件进行测试

4. IDEA 中使用 Git

4.1 配置 Git

找到 File -> Settings -> Version Control -> Git，指定 Git.exe 的路径，如果 Git 是在默认安装路径下的，一般都会自动识别，然后点击 Test 测试 Git 是否可以正常工作，出现版本号则说明正常。

4.2 创建工程初始化本地库

创建一个工程，用 Git 进行管理

创建好工程和内容之后，点击 VCS -> Import into Version Control -> Create Git Repository。打开当前工程文件夹，如果出现了一个 .git 文件夹就说明本地库初始化成功。（注意：.git 文件夹是隐藏的，需要在文件管理器中开启显示隐藏内容）

同时工程中的文件就会褐色，表示这些内容还没有被 Git 跟踪。

4.3 忽略 IDEA 特定文件和文件夹

我们只用 Git 管理工程的文件内容，对于 IDEA 生成的内容不做管理，所以需要 Git 忽略 IDEA 特定文件和文件夹。

打开 当前工程文件夹/.git/info/exclude 文件，将需要忽略的IDEA 特定文件和文件夹列在里面。

4.3 添加到暂存区

工程文件夹上右键 -> Git -> Add 将工程目录下所有内容添加到暂存区，此时之前为褐色的就变成蓝色了，说明这些内容已经被 Git 追踪了。

4.4 提交到本地库

可以右键提交，也可以点击工具栏上的图标进行提交，输入 commit message（随便输入），点击 commit

此时关于 Git 的一些本地操作就基本完成了，如果需要将代码托管到 GitHub 远程仓库进行管理的话，需要先登录 GitHub 创建一个远程仓库，将远程仓库和当前工程进行关联。

4.5 配置远程仓库地址并将本地代码推送到远程仓库

首先先复制远程仓库的地址，如果没有配置 SSH 免密登录的话就先复制 HTTP 的地址

点击 OK -> 点击 Push 即可，稍等片刻，访问国外网站会比较慢。

等到提示 Push 成功，现在到 GitHub 远程仓库刷新一下，本地工程代码就上传上去了。

4.6 拉取远程仓库

如果本地工程内容不是最新版，则需要从远程仓库拉取最新内容

点击 pull 即可，如果有冲突，则还要解决冲突。

4.7 克隆仓库

点击 clone 即可。
