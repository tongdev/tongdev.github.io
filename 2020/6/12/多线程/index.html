Java 多线程

Java 多线程总结篇，创建多线程、线程同步、线程通信。

多线程

使用多线程的优点

  1. 提高程序执行的效率
  2. 提高 CPU 利用率
  3. 改善程序结构。有多个任务时，将一个任务交给一个线程去独立执行，利于理解和修改。

程序、进程、线程的概念

程序：是为了完成特定的任务使用某种编程语言而编写的，是一段静态的代码。

进程：是资源分配的基本单位，是正在运行的一个程序，它是动态的，有自己的生命周期。

线程：是CPU调度的基本单位，共享进程的资源，是程序的一条执行路径。

线程的生命周期

线程有五种状态，分别是：新建、就绪、运行、阻塞、死亡

线程状态关系图

线程的分类

线程可以分为用户线程和守护线程

  • 它们几乎是相同的，唯一的区别是判断 JVM 何时离开。
  • 守护线程是用来服务用户线程的，在调用 start() 之前，使用 setDaemon(true) 可以把一个用户线程变成一个守护线程。
  • 垃圾回收线程、检测异常线程等都是守护线程。
  • 若 JVM 中都是守护线程，当前 JVM 将退出。

线程相关的方法

  • run()：将线程要执行的操作写到方法中
  • start()：启动线程，执行 run()
  • yeild()：让出 cpu 执行权
  • join()：a.join(b) 阻塞 a 线程，直到 b 线程执行结束
  • setName()：设置线程的名称
  • getName()：获取当前线程的名称
  • currentThread()：静态方法，返回当前正在执行的线程
  • sleep(long millisecond)：让当前线程睡眠指定的时间
  • stop()：手动使当前线程死亡，方法已过时。
  • isAlive()：判断当前线程是否存活，返回值 true 或 false

线程优先级相关

  • 优先级常量

    MAX_PRIORITY = 10 最大优先级

    MIN_PRIORITY = 1 最小优先级

    NORM_PRIORITY = 5 默认优先级

  • setPriority(int priority)：设置线程优先级

  • getPriority(int priority)：获取线程优先级

  • 说明

    线程创建时继承父线程的优先级

    低优先级只是获得调度的概率低，并非一定是在高优先级线程之后才被调用

线程通信相关

  • wait()：阻塞当前线程
  • notify()：唤醒一个被阻塞的线程，如果有多个，则唤醒优先级高的那个。
  • notifyAll()：唤醒所有被阻塞的线程

创建多线程的方式

方式一和方式二是 JDK5.0 之前的

1. 继承 Thread 类

2. 实现 Runnable 接口

继承 Thread 类和实现 Runnable 接口比较

比较创建多线程的方式一和方式二

  • 开发中优先使用实现 Runnable 接口的方式
  • 原因：
    1. 实现 Runnable 接口的方式没有单继承的限制
    2. 实现的方式更适用于多个线程有共享数据的时候（具体表现为：创建多个线程时传入一个实现类对象，同步监视器使用 this，共享数据变量不需要 static）
  • 相同点：两种方式最终都是实现了 Runnable 接口重写了 run()，Thread 类也是实现了 Runnable 接口。

方式三和方式四是 JDK5.0 之后新增的

3. 实现 Callable 接口

Runnable 和 Callable 比较

与 Runnable 相比，Callable 更强大一些

  • 相比于 run()，call() 有返回值
  • 可以抛出异常
  • 支持泛型
  • 可以借助 FutureTask 对执行结果进行操作，比如通过 get() 获取 call() 的返回值

4. 使用线程池

实现线程池的优点：

  1. 提高响应速度（减少了创建新线程的时间）
  2. 降低资源消耗（重复利用线程池中的线程，不需要每次都创建）
  3. 便于线程管理

线程同步机制

概述

  1. 为什么需要线程同步机制？

    首先，线程同步机制是为了解决线程安全问题。主要有两种方式：synchronized（同步代码块、同步方法）、Lock。

  2. 线程安全问题是如何产生的？

    条件：多线程执行环境下

    原因：多个线程共同操作共享数据（共同操作的变量）而产生的线程安全问题

  3. 使用线程同步机制后，当执行到被同步的代码时，只能有一个线程执行，其他线程等待，单线程执行。

同步监视器的使用

同步监视器，俗称锁，任何对象都可以充当锁，往往根据实际情况来选定。

同步代码块

  • 使用实现 Runnable 接口的方式创建多线程时，同步监视器可以考虑使用 this。但必须保证 this 唯一。

  • 使用继承 Thread 类的方式，同步监视器慎用 this，可以考虑使用当前类的字节码对象。

同步方法

  • 同步方法仍然需要同步监视器，只是不需要显示声明。
  • 静态的同步方法，同步监视器为：当前类的字节码对象，Class.class
  • 非静态的同步方法，同步监视器为：this

同步代码块和同步方法的比较

相较于同步方法，使用同步代码块更加灵活，只要用 synchronized 包起来即可，同步方法只能作用于方法上，有局限性。

Lock

JDK5.0 新增的同步机制

Lock 是一个接口，使用时创建它的实现类 ReentrantLock

加锁：lock()，解锁：unlock()

synchronized 和 Lock 的异同

同：都可以解决线程的安全问题

异：synchronized 方式是同步代码块执行完后自动释放锁，而 Lock 方式是手动加锁手动解锁，相较于 synchronized 方式更加灵活。

三种方式使用优先顺序：Lock > 同步代码块 > 同步方法

线程的死锁问题

死锁

不同的线程分别占用对方需要的资源不放弃，都在等待对方放弃自己需要的同步资源，就形成了死锁。

出现死锁后，不会出现异常，只是所有的线程都处于阻塞状态，无法继续执行。

解决方法

  1. 使用使用专门的算法或规则避免死锁
  2. 尽量减少同步资源的定义
  3. 尽量避免嵌套同步

线程通信

线程通信的三个方法：

  1. wait()：执行完此方法后，当前线程进入阻塞状态。

  2. notify()：执行完此方法后，将唤醒一个被阻塞的线程，如果有多个被阻塞的线程，则唤醒优先级高的那一个。

  3. notifyAll()：执行完此方法之后，将唤醒所有被阻塞的线程。

说明：

  1. wait()、notify()、notifyAll()这三个方法必须在同步代码块或同步方法中使用。
  2. wait()、notify()、notifyAll()这三个方法的调用者必须和同步监视器保持一致，否则会报异常。
  3. wait()、notify()、notifyAll()这三个方法定义在 Object 类中。

面试题：比较 sleep() 和 wait()

  • 相同点：sleep() 和 wait() 都会使当前线程进入阻塞状态
  • 不同点
    1. 方法定义位置不同：sleep() 定义在 Thread 类中，wait() 定义在 Object 类中
    2. 使用范围不同：sleep() 随处可用，而 wait() 只能用于同步代码块或同步方法中
    3. 关于是否释放锁：如果两个方法都使用在同步代码块或同步方法中，sleep() 阻塞线程后不会释放锁，而 wait() 阻塞线程后会释放锁，供其他线程使用。
